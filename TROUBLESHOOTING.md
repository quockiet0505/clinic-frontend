# Troubleshooting & Deployment Notes

This document logs critical issues encountered during deployment and the permanent solutions implemented.

## Issue: 502 Bad Gateway on `patient.duongquockiet.id.vn`
**Date:** 2026-07-12  
**Symptom:** The website returned a 502 Bad Gateway error via Cloudflare.

### Root Causes
1. **Docker Swarm Registry Authentication Expiry:**
   * Frontend services (`clinic_patient-web`, `clinic_admin-web`) are pulled from Google Artifact Registry (`asia-southeast1-docker.pkg.dev`).
   * The CI/CD deployment workflow used a temporary OIDC access token (OAuth2 access token) generated by GitHub Actions to authenticate Docker on the VM via `docker login`, and updated the services using `docker service update --with-registry-auth`.
   * The token expires after 1 hour. When Docker Swarm tried to reschedule a container or pull a fresh image after this window, it failed with:
     ```
     error from registry: authentication failed
     ```
2. **MySQL Database Crash:**
   * The database container `clinic_mysql-db` had entered a `Dead` state on the VM.
   * This caused the backend `clinic_backend` to fail to connect with:
     ```
     java.net.UnknownHostException: mysql-db / Communications link failure
     ```
   * Since both the frontend and backend services were down, the Nginx Proxy could not reach them, returning a 502 error to Cloudflare.

---

### Permanent Fix

#### 1. Configure Standalone GCR Credential Helper on VM
To prevent registry authentication from expiring, we installed the native, standalone Google GCR Docker credential helper (`docker-credential-gcr`) on the VM.
* **Installation:** Downloaded `docker-credential-gcr` version 2.1.29 and moved it to `/usr/bin/docker-credential-gcr`.
* **Configuration:** Configured the root user's Docker configuration (`/root/.docker/config.json`) to use the helper for Artifact Registry:
  ```json
  {
      "auths": {
          "asia-southeast1-docker.pkg.dev": {}
      },
      "credHelpers": {
          "asia-southeast1-docker.pkg.dev": "gcr"
      }
  }
  ```
Since the VM runs with an attached GCP Compute Engine Service Account (`compute-vm-sa@clinicqa-500817.iam.gserviceaccount.com`), the helper dynamically requests fresh OAuth tokens from the VM's Metadata Service whenever Docker needs to pull an image. This access never expires.

#### 2. Update CI/CD Workflows
We updated the `.github/workflows/deploy.yml` files in both the frontend and backend repositories:
* Removed the temporary `docker login` commands on the VM.
* Removed the `--with-registry-auth` flag from the `docker service update` and `docker stack deploy` commands.
This forces Docker Swarm to use the VM's local Docker config (which uses the `gcr` credential helper) instead of passing and saving temporary tokens from the CI/CD runner.

### Services Status Reference
To check service status and logs on the VM:
```bash
# SSH into VM
gcloud compute ssh clinic-vm --zone "asia-southeast1-a" --tunnel-through-iap --project "clinicqa-500817"

# Inspect Swarm Services
sudo docker service ls
sudo docker service ps clinic_patient-web

# Get Container logs (GCP Logs are used, so local logs might be forwarded)
sudo docker service logs clinic_backend
```
